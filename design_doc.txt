StepGuard: Iterative Process Monitoring and Regression-Aware Suffix Repair for Improving LLM Reasoning

StepGuard is a research pipeline for improving LLM reasoning by monitoring step-by-step reasoning traces, identifying the earliest risky step, and applying targeted suffix repair. The current framework focuses on GSM8K math reasoning, but the design leaves room for later evidence-based datasets and additional process-supervision signals.

The central idea is that reasoning failures should be detected within the reasoning process, not only at the final answer. StepGuard scores each reasoning step, finds the first step whose risk exceeds a threshold, preserves the prefix before that step, and regenerates the suffix. Candidate repairs are then accepted or rejected according to an acceptance policy.


Proposal

LLMs often produce reasoning steps that look plausible but contain:
- unsupported assumptions
- invalid logical jumps
- contradictions
- logical gaps

Rather than evaluating only the final answer, StepGuard evaluates the reasoning process step by step. The current framework supports:
- trace generation
- step splitting
- verifier scoring with an LLM judge
- contradiction scoring with local RoBERTa-MNLI
- per-step risk scoring
- earliest risky step detection
- iterative suffix repair
- multiple repair acceptance modes
- evaluation of answer correctness, process risk, repair behavior, and proxy detection metrics


Current Pipeline

1. Generate a reasoning trace and final answer.
2. Split the trace into numbered reasoning steps.
3. Score each step with:
   - verifier score: LLM judge p_wrong per step
   - contradiction score: RoBERTa-MNLI contradiction probability against recent prior steps
   - evidence support: reserved for future non-math/evidence-based tasks
4. Combine scores into a per-step risk value.
5. Find the earliest risky step: the first step whose risk exceeds the risk threshold.
6. Repair by preserving the prefix before the earliest risky step and regenerating the suffix.
7. Score the candidate repaired trace.
8. Apply a repair acceptance policy.
9. Iterate until there is no risky step, risk worsens, improvement plateaus, or max_iters is reached.
10. Evaluate final answer correctness, risk reduction, repair behavior, and detection metrics.

Current default experimental flow:

generate -> iterative-repair -> evaluate

The older score-traces and repair-traces commands remain available as single-pass/baseline commands.

CLI Commands

prepare-dataset:
Exports a dataset into JSONL format for pipeline use.

generate-traces:
Reads a JSONL dataset and generates a reasoning trace plus final answer for each question. Writes generated.jsonl.

score-traces:
Reads generated.jsonl, splits traces into steps, computes verifier and contradiction scores, computes per-step risk, and finds the earliest risky step. Writes scored.jsonl. This is useful for single-pass or diagnostic runs.

repair-traces:
Reads scored.jsonl and performs one suffix repair from the earliest risky step. Writes repaired.jsonl. This is a single-pass baseline command.

iterative-repair:
Reads generated.jsonl, scores each trace, repairs from the earliest risky step, scores the candidate repair, applies the configured repair acceptance policy, and repeats until stopping. Writes iterative_repaired.jsonl.

evaluate:
Reads iterative_repaired.jsonl by default, falling back to repaired.jsonl if needed. Computes correctness, risk, repair-loop, and proxy repair-detection metrics. Writes metrics.json, evaluation_records.jsonl, and manifest.evaluate.json.

run-pipeline:
Runs:

generate-traces -> iterative-repair -> evaluate


Scoring

Contradiction score:
For each step i, the premise is the previous one or two reasoning steps. The hypothesis is the current step. RoBERTa-MNLI returns a contradiction probability. Step 0 receives contradiction score 0.0 because there is no prior step to contradict.

Range:
- 0 means no contradiction detected.
- 1 means strong contradiction detected.

Verifier score:
For each trace, the LLM judge receives the question and full list of reasoning steps. It returns p_wrong for each step.

Range:
- 0 means the step is judged valid.
- 1 means the step is judged invalid.

Current risk score:

risk[i] = min(1.0, 0.75 * verifier[i] + 0.25 * contradiction[i])

Earlier experiments used:

risk[i] = min(1.0, verifier[i] + contradiction[i])

The current verifier-heavy formula was introduced because the additive formula with a low threshold over-triggered repairs and caused correct-to-wrong regressions.

Earliest risky step:

earliest_bad_step = first step i where risk[i] > risk_threshold

If no step exceeds the threshold, earliest_bad_step is null and the trace is not repaired.


Repair Acceptance Modes

StepGuard now separates repair generation from repair acceptance. A candidate repair can reduce measured risk while still making the final answer worse, so acceptance is handled by a configurable policy.

risk_only:
Baseline behavior. Accepts a candidate repair if average risk does not worsen.

Acceptance rule:

new_avg_risk <= old_avg_risk

oracle_guard:
Research-only upper-bound mode. Uses the gold answer to reject repairs that would turn a correct original answer into an incorrect final answer.

Acceptance rule:

risk improves
and not(original answer correct and candidate answer incorrect)

This is not deployable because it uses gold labels inside the repair loop. It is useful for estimating how much improvement would be possible with a perfect regression detector.

judge_guard:
Deployable regression-aware mode for OpenAI runs. Calls an LLM repair acceptance judge after a risk-improving candidate repair. The judge compares the current trace/answer with the candidate trace/answer and returns support and regression-risk scores.

Acceptance rule:

risk improves
and repaired_answer_support >= original_answer_support - support_tolerance
and regression_risk <= max_regression_risk

The judge also returns prefer_repaired, but this is logged as diagnostic information and is not required for acceptance. This makes judge_guard a regression guard rather than a strict "must be better" judge.


Iterative Repair Loop

current_trace = original model_trace
current_answer = original model_answer

Score current_trace
Record original risk summary

For iter from 1 to max_iters:
    If there is no earliest risky step:
        stop with reason "no_bad_step"

    k = earliest risky step
    prefix = steps before k

    Generate candidate suffix starting at step k + 1
    candidate_trace = prefix + candidate suffix

    Score candidate_trace

    old_avg_risk = average risk across current steps
    new_avg_risk = average risk across candidate steps
    improvement = old_avg_risk - new_avg_risk

    Apply repair acceptance mode:
        risk_only
        oracle_guard
        judge_guard

    If candidate is rejected:
        stop with acceptance reason

    If candidate is accepted:
        current_trace = candidate_trace
        current_answer = candidate_answer
        final_score = candidate_score

    If improvement < improvement_threshold:
        stop with reason "improvement_below_threshold"

If max_iters is reached:
    stop with reason "max_iters"

Write final trace, final answer, final scores, final risks, and iteration logs.


Evaluation

For each record:
- Compare original answer to gold answer.
- Compare final answer to gold answer.
- Categorize answer outcome:
  - wrong_to_correct
  - wrong_to_wrong
  - correct_to_correct
  - correct_to_wrong
- Compute risk delta:
  - original_avg_risk - final_avg_risk
- Count repair-loop behavior:
  - stop reason
  - repair attempts
  - accepted repair records
  - accepted repair iterations

metrics.json includes:
- correctness:
  - original_correct
  - final_correct
  - original_accuracy
  - final_accuracy
  - accuracy_delta
  - answer outcome counts
- risk:
  - mean original/final average risk
  - mean risk delta
  - mean original/final max risk
  - mean original/final number of risky steps
- repair_loop:
  - stop reason counts
  - repair attempt rate
  - accepted repair record rate
  - accepted repair iteration rate
  - mean iteration logs per record
- repair_detection:
  - attempted-repair precision, recall, F1
  - accepted-repair precision, recall, F1

The current repair-detection precision/recall/F1 metrics are example-level proxy metrics:

positive prediction = repair attempted or accepted
actual positive = original answer was wrong

These are not first-bad-step precision/recall/F1. True first-bad-step metrics would require manually labeled reasoning traces.


Current Experimental Results

The current 100-example GSM8K experiments use the same generated traces for the calibrated comparison runs.

Run summary:

dev_50:
- Original aggressive risk-only baseline.
- Risk formula: min(1.0, verifier + contradiction)
- Threshold: 0.10
- Final accuracy: 0.97
- Accuracy delta: -0.02
- Correct-to-wrong regressions: 2
- Repair attempt rate: 0.96
- Accepted repair rate: 0.46
- Mean risk delta: 0.0198

dev_50_2:
- Calibrated risk-only baseline.
- Risk formula: 0.75 verifier + 0.25 contradiction
- Threshold: 0.20
- Final accuracy: 0.98
- Accuracy delta: -0.01
- Correct-to-wrong regressions: 1
- Repair attempt rate: 0.14
- Accepted repair rate: 0.11
- Mean risk delta: 0.0023

dev_50_3:
- Calibrated risk formula plus oracle_guard.
- Final accuracy: 0.99
- Accuracy delta: 0.00
- Correct-to-wrong regressions: 0
- Repair attempt rate: 0.15
- Accepted repair rate: 0.10
- Mean risk delta: 0.0043

dev_50_4:
- Initial strict judge_guard requiring prefer_repaired.
- Final accuracy: 0.99
- Accuracy delta: 0.00
- Correct-to-wrong regressions: 0
- Repair attempt rate: 0.14
- Accepted repair rate: 0.00
- Mean risk delta: 0.0000
- Interpretation: protected accuracy but rejected all repairs because prefer_repaired was too strict.

dev_50_5:
- Updated judge_guard that uses answer support and regression risk, with prefer_repaired logged only as diagnostic.
- Final accuracy: 0.99
- Accuracy delta: 0.00
- Correct-to-wrong regressions: 0
- Repair attempt rate: 0.14
- Accepted repair rate: 0.09
- Mean risk delta: 0.0025

Interpretation:
The aggressive baseline reduced measured risk the most but over-intervened and caused answer regressions. The calibrated risk formula reduced repair frequency and reduced regressions. The oracle guard showed that regression-aware acceptance can preserve original accuracy. The updated judge guard approximated this behavior without using gold answers: it preserved final accuracy, avoided correct-to-wrong regressions, and still accepted low-regression-risk repairs that reduced measured process risk.


Research Framing

Current claim:
StepGuard has a working end-to-end pipeline for process-level reasoning monitoring and targeted suffix repair. Early experiments show that both risk calibration and repair acceptance policy strongly affect the tradeoff between process-risk reduction and final-answer stability.

Important limitation:
Current results are preliminary and use a 100-example GSM8K subset. The repair-detection precision/recall/F1 metrics use final-answer correctness as a proxy label, not human-labeled hallucination or first-bad-step annotations.

Current strongest non-oracle setting:
Verifier-heavy risk scoring with threshold 0.20 and judge_guard acceptance.
